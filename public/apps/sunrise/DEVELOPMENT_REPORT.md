# Sunrise - 製造データ分析ダッシュボード 開発報告

> **ビジョン：製造現場のデータを即座に可視化し、品質改善と効率化を加速する**

---

## 1. 現在の課題（Problem）

製造現場のデータ分析業務で日常的に発生している課題を整理しました。

### 1-1. 複数システムに散在するデータの集約困難

製造データが複数のシステム（Snowflake、ローカルDB、外部システム）に分散しており、統合的な分析が困難な状況です。各システムからデータを手動で取得し、Excelで集計する作業に多大な時間を要しています。

**具体的な問題点：**
- データ取得に1回あたり数分～数十分かかる
- 手動集計によるヒューマンエラーのリスク
- リアルタイムな状況把握ができない

### 1-2. 品質指標の可視化と分析の負担

直行率、CPK、NG項目などの品質指標を分析する際、以下の課題がありました：

```
現状の業務フロー:
├── データ抽出（SQL手動実行）  ← 時間がかかる
├── Excelへのコピー＆ペースト  ← 手作業
├── グラフ作成（手動）         ← 属人化
└── → 分析に着手するまでに数時間かかる
```

**その結果：**
- 分析頻度が低下し、問題発見が遅れる
- 設備・機種・時間帯別の詳細分析が困難
- トレンド把握や異常検知が後手に回る

### 1-3. 階層的なデータ構造の管理

拠点 → 工程分類 → ライン → 検査工程 → 設備という階層構造を持つ製造データの管理が煩雑で、フィルタリングや検索に時間がかかっていました。

## 2. Sunriseでこう変わる（Solution）

| # | 現状（BEFORE） | 導入後（AFTER） |
|---|---|---|
| 1 | 複数システムからの手動データ取得（数十分） | **統合データ取得**で数秒で完了 |
| 2 | Excelでの手動集計とグラフ作成 | **自動集計・可視化**でワンクリック分析 |
| 3 | 静的なExcelレポート | **インタラクティブなダッシュボード**でドリルダウン分析 |
| 4 | 分析結果の共有が困難 | **Webベース**でどこからでもアクセス可能 |
| 5 | データ取得の度に接続待ち（2秒/回） | **接続プール**で0.05秒に短縮（40倍高速化） |
| 6 | 過去分析の再実行に時間がかかる | **Redisキャッシュ**で即座に結果表示（今後実装予定：長期保存） |

### 導入後の構成イメージ

```
Sunrise統合ダッシュボード
└── データソース切り替え
    ├── Snowflake（Hint）
    │   ├── HF1R6M01（検査結果）
    │   ├── HF1RDM01（詳細データ）
    │   └── HF1REM01（拡張データ）
    └── ローカルDB（MySQL/SQLite）
        └── 階層管理
            ├── 拠点（Site）
            ├── 工程分類（MajorProcess）
            ├── ライン（Line）
            ├── 検査工程（InspectionProcess）
            └── 設備（Equipment）
```

**補足：** 階層的なフィルタリングにより、目的のデータに素早くアクセスできます。設備使用履歴も記録され、よく使う設備が優先表示されます。

## 4. 主要機能の詳細

### 4-1. 直行率分析

号機・機種・時間帯別の直行率を多角的に分析できます。

```
入力: 拠点、工程、ライン、検査工程、日付範囲
  ↓
処理: HF1R6M01テーブルからデータ取得
      MK_DATEでユニーク化（重複排除）
      号機/機種/プログラム別に集計
  ↓
出力: 
  - 号機毎の直行率グラフ（棒グラフ＋折れ線）
  - 機種毎の直行率グラフ
  - 時間帯毎の直行率グラフ
  - 階層的なデータテーブル（号機→機種→プログラム）
```

**特徴：**
- OK/NG件数と直行率を同時表示
- 階層的なドリルダウン（号機→機種→プログラム）
- 時系列トレンド分析

### 4-2. CPK/ヒストグラム分析

工程能力指数（CPK）を計算し、測定値の分布をヒストグラムで可視化します。

```
入力: 検査項目の選択
  ↓
処理: MEASUREカラムのデータを取得
      号機・機種・プログラム別にグループ化
      統計値計算（平均、標準偏差、中央値）
      CPK計算（Cp, Cpk, Cpu, Cpl）
  ↓
出力:
  - 重ね合わせヒストグラム（グループ別に色分け）
  - ボックスプロット（外れ値表示）
  - 規格上下限線（赤色破線）
  - CPK値と判定（良好/要注意/改善必要）
```

**CPK判定基準：**
- CPK ≥ 1.67: 良好（緑）
- 1.33 ≤ CPK < 1.67: 要注意（オレンジ）
- CPK < 1.33: 改善が必要（赤）

### 4-3. NG集計（パレート図）

NG項目を発生件数順に並べ、累積発生率を表示します。

```
入力: フィルター条件
  ↓
処理: NG項目別に集計
      発生件数の多い順にソート
      累積発生率を計算
  ↓
出力: パレート図
  - 棒グラフ：NG項目別発生件数
  - 折れ線グラフ：累積発生率
  - 80-20の法則による重点項目の特定
```

**活用方法：**
累積発生率80%に達するまでの上位項目に対策を集中することで、効率的に全体の不良率を削減できます。

### 4-4. 検査タクト分析

検査にかかった時間（タクトタイム）を分析し、生産効率を評価します。

```
入力: フィルター条件
  ↓
処理: タクトタイムデータを取得
      日時・号機・機種別に集計
  ↓
出力: タクトタイムテーブル
  - 日時、号機、機種、タクトタイム、判定
```

## 5. Sunrise固有の構造・連携

### データ構造の階層管理

```
[ローカルDB]                    [Snowflake (Hint)]
┌──────────────────┐           ┌──────────────────┐
│ Site（拠点）      │           │ HF1R6M01         │
│  ↓                │           │  - STA_NO1/2/3   │
│ MajorProcess      │  ──連携→  │  - PARTSNAME     │
│  ↓                │           │  - PROGRAM_NAME  │
│ Line（ライン）    │           │  - MEASURE       │
│  ↓                │           │  - OPEFIN_RESULT │
│ InspectionProcess │           └──────────────────┘
│  ↓                │
│ Equipment（設備） │
│  ↓                │
│ ExternalMapping   │
│  - STA_NO1/2/3    │
└──────────────────┘
```

**連携の仕組み：**
1. ユーザーがUIで階層を選択（拠点→工程→ライン→検査工程）
2. 選択された設備に紐づくSTA_NO1/2/3を取得
3. SnowflakeのHF1R6M01テーブルをSTA_NO1/2/3でフィルタリング
4. 取得したデータをキャッシュに保存
5. 各種分析機能で利用

### 接続プールによる高速化

```
従来の接続方式:
  リクエスト毎に接続 → 認証 → クエリ → 切断
  ⏱️ 約2秒/回

接続プール方式:
  事前に接続を確立 → プールから取得 → クエリ → プールに返却
  ⏱️ 約0.05秒/回（40倍高速化）
```

**設定：**
- 最大接続数: 5
- 最小接続数: 1（事前作成）
- アイドルタイムアウト: 300秒
